## Context

The current Roger MVP successfully routes voice or typed instructions through local wake/VAD/STT/TTS components and pi-agent RPC sessions. Verification identified the remaining gaps as hardening concerns rather than blockers for the MVP: offline fallback is manual (`--offline`), streamed pi RPC events are collected only as final text in the runner, local TTS exceptions can still affect completion handling, active tasks cannot be stopped from Roger, and routing rules are still mostly hard-coded.

Relevant research from the current code and pi docs:

- Pi RPC supports provider/model selection at startup with `--provider <name>` and `--model <pattern>`, so Roger can keep online mode as pi default and use `--provider llama-cpp --model <configured>` for offline fallback.
- Pi also has `--offline` / `PI_OFFLINE=1` to disable pi startup network operations such as update checks and telemetry. This is separate from choosing a llama.cpp work model and should be considered when launching fallback sessions.
- Pi RPC accepts `abort`, `abort_bash`, and `abort_retry` commands, so in-flight cancellation is technically available through the same JSONL channel Roger already uses.
- Pi RPC streams `message_update`, `tool_execution_start`, `tool_execution_update`, `tool_execution_end`, `agent_end`, retry, queue, and error events. `tool_execution_update.partialResult` is accumulated output, which is useful for a replace-in-place log surface.
- Roger already has a small `TaskLog` abstraction, an overlay feedback surface, and CLI/daemon paths where lifecycle events can be wired without changing speech backends.

## Goals / Non-Goals

**Goals:**

- Detect online provider failure or explicit offline state and choose the correct pi launch mode automatically.
- Surface pi-agent progress in a visible textual log while keeping TTS concise.
- Make TTS best-effort: speech failures must be visible but must not turn a successful pi task into a failed task.
- Add user-facing cancellation for active work, using pi RPC abort commands and honest reporting.
- Make the router extensible enough for future domains without coupling new domains to the voice pipeline.

**Non-Goals:**

- Replace pi-agent execution with Roger-native task execution.
- Make STT/TTS cloud-backed; speech remains local/offline.
- Build a full GUI task manager; the next step only needs a reliable log surface and overlay integration.
- Guarantee every provider outage can be detected before dispatch; fallback may also trigger after prompt/session failure.
- Implement advanced semantic intent classification unless needed later; a registry/rules layer is enough for this follow-up.

## Decisions

### Use a model availability policy instead of scattering `--offline` checks

Create a small policy component responsible for deciding `online`, `offline-fallback`, or `unavailable`. It should combine explicit CLI config, optional connectivity/provider probes, and observed pi RPC startup/prompt failures. The runner should consume the decision rather than independently deciding command-line flags.

Alternatives considered:

- Keep only the current `--offline` flag: simple, but does not satisfy automatic fallback.
- Always use llama.cpp first: works offline, but violates the preference to use pi's configured online default when available.

### Treat llama.cpp fallback as a pi launch profile

Offline fallback should launch pi with llama.cpp provider/model arguments, and should consider `PI_OFFLINE=1` or pi's `--offline` startup flag to avoid unrelated network startup checks. The exact command should remain generated by `PiSessionManager` so sessions continue to use the correct cwd and session directory.

Alternatives considered:

- Call llama.cpp directly from Roger: would bypass pi tools, sessions, and terminal-equivalent behavior.
- Use a separate offline executor: increases divergence from pi permissions and session behavior.

### Stream every pi RPC event into a task log sink

`PiAgentRunner` should expose event callbacks or accept a `TaskLog`/observer. The runner can still return final collected text, but it must not discard intermediate events. The log should store structured events and render concise summaries for overlay/CLI while retaining details for debugging.

Alternatives considered:

- Only show final text: current behavior, insufficient for long tasks.
- Speak every event: noisy and conflicts with concise spoken summaries.

### Wrap TTS behind a safe speaker boundary

Introduce a safe speaker wrapper or make `SynthesizingSpeaker.speak()` failure-aware. Callers should receive a success/failure signal or logged exception, while task completion remains based on pi/router result. The overlay/text result should remain the source of truth.

Alternatives considered:

- Let daemon `on_error` catch TTS failures: too late; it can misclassify a completed task as a failed cycle.
- Disable TTS by default: conflicts with desired voice response behavior.

### Map natural stop intent to explicit active-task cancellation

Roger should recognize stop/cancel phrases while a task is active and send pi RPC `abort`. If a bash command is specifically running or retry delay is active, Roger may also attempt `abort_bash` or `abort_retry`. It should report whether the abort command was accepted and whether the agent ended with an aborted/error reason.

Alternatives considered:

- Terminate the pi subprocess always: simpler but may lose session state and cleanup opportunities.
- Ignore stop requests until the next wake cycle: poor UX for long-running or risky tasks.

### Make routing rules data-driven before adding new domains

Keep the current rule-based approach, but move domain keywords, safe/destructive ambiguity rules, and domain metadata into registry/config objects. This allows adding named projects, communications, or media domains without changing wake/VAD/STT/TTS code.

Alternatives considered:

- Full LLM-based router: flexible, but adds latency and model dependency to routing.
- Hard-code more keyword lists: quick, but continues to grow technical debt.

## Risks / Trade-offs

- **False offline detection** → Prefer online by default, use bounded probes, and only fallback automatically on known connectivity/provider errors or explicit offline mode.
- **llama.cpp fallback has weaker model/tool performance** → Make offline mode visible in overlay/logs and document model expectations.
- **Streaming logs can grow large** → Store structured events with truncation/rotation and keep TTS summaries capped.
- **Abort semantics may vary by active operation** → Report command acceptance separately from final stopped state, and fall back to subprocess termination only as a last resort.
- **TTS failure loops in daemon** → Rate-limit repeated TTS failure messages and keep textual overlay active.
- **More configurable routing can become opaque** → Add `roger route --explain` or testable rule explanations when extending the registry.

## Migration Plan

1. Add tests around the current warning cases: automatic fallback, streamed event logging, TTS failure isolation, abort command support, and configurable routing.
2. Introduce new abstractions behind current constructors with defaults preserving existing CLI behavior.
3. Wire typed `roger task` first because it removes speech uncertainty, then wire `listen-once` and daemon.
4. Update setup docs with offline fallback behavior, cancellation phrases, and log locations.
5. Keep rollback simple: disable automatic fallback with config, use `--no-tts` if speech fails, and keep existing `--offline` as an explicit override.

## Open Questions

- Should `gemma4` remain the default llama.cpp fallback model for real tool-using laptop tasks, or should a coding-tuned GGUF replace it?
- Should Roger probe only internet reachability, pi provider health, or both before choosing fallback?
- Where should persistent task logs live: `.roger/logs`, per-session folders, or both?
- Should cancellation be triggered only by a new wake phrase plus stop command, or also by a global hotkey/CLI command?
